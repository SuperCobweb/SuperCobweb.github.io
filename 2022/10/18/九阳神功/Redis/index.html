Redis

发表于2022-10-18|更新于2023-04-03|九阳神功
|阅读量:

基础知识

redis是一个key-value的NoSQL数据库，定期通过异步操作将数据库数据flush到硬盘保存，是已知最快的key-value数据库，支持多钟数据结构，单个value最大限制为1GB，读写性能达到10万/秒。

Redis的主要缺点是数据库容量受到物理内存的限制。不能用作海量数据的高性能读写，因此Redis适合的场景主要局限在较小数据量的高性能操作和运算上。

Redis为什么将所有数据都放到内存中？

因为Redis为了达到更快地读写速度，Redis会通过异步的方式将数据写入磁盘，所以Redis具有快速和持久化的特征。如果不放到内存中，磁盘的I/O会严重影响到Redis 的性能。如果设置了最大使用内存，则已有数据记录达到最大内存限制后则不能继续插入新值。

有MySQL不就够了吗，为什么还要使用Redis？

高性能：我们用MySQL查询数据时，需要从磁盘进行查找，速度是比较慢的，如果我们经常对该值进行查找，则会一遍一遍的请求MySQL数据库，但是如果我们将其存入缓存，再次请求时，因为是直接访问内存，速度是极快，当数据库的内容发生改变时，只需改变缓存的值即可。

高并发：直接操作缓存能够承受的请求是远远大于直接访问数据库的，所以我们可以将部分值存到缓存中去，当用户请求时直接从缓存中获取即可。

Redis 可以用来做什么？

  • Redis 常用来做缓存，作为分布式缓存的中间件
  • Redis 可以作为数据库，实现诸如点赞、关注、排行等对性能要求极高的互联网需求
  • Redis 可以用来作为计算工具，能用很小的代价统计诸如PV、UV、用户在线天数等数据
  • 其他使用场景：实现分布式锁、可以作为消息队列使用

Redis 是单线程，为什么还这么快？

  1. 对服务端程序来说，线程的切换和锁通常是性能杀手，单线程避免了线程切换和竞争带来的消耗
  2. Redis 的大部分操作都是在内存上完成的，这是实现高性能的真要原因。
  3. Redis 采用IO多路复用机制，在网络IO操作中能够并发处理大量的客户端请求，实现高吞吐量
  4. Redis 是单线程主要是指Redis的网络IO和键值对读写是由一个线程来完成的，而Redis的其他功能，如持久化、异步删除、集群数据同步则是依赖其他线程来执行的

Redis数据类型

  • 5种核心的数据类型：字符串、列表、哈希、集合、有序集合
  • 复杂数据类型：Bitmap、HyperLogLog、Geo
  • Redis 在5.0中新增了Streams数据类型，他是一个消息队列

哈希字典：（包含键值对的无序散列表）用户信息，商品信息

集合：共同好友、共同关注

有序集合：排行榜

Redis 底层数据结构

共有6种数据结构：简单动态字符串、双向链表、压缩列表、字典、跳表、整数数组

简单动态字符串SDS：

SDS 相比C 字符串的优势：

  • SDS保存了字符串的长度，而C字符串不保存长度，需要遍历整个数组（找到’\0’为止）才能取到字符串长度。
  • 修改SDS时，检查给定SDS空间是否足够，如果不够会先拓展SDS 的空间，防止缓冲区溢出。C字符串不会检查字符串空间是否足够，调用一些函数时很容易造成缓冲区溢出（比如strcat字符串连接函数）。
  • SDS预分配空间的机制，可以减少为字符串重新分配空间的次数。
    • 当字符串长度小于1M时，每次扩容都是加倍现有的空间
    • 如果字符串长度超过1MB时，每次扩容只会扩展1MB的空间

一个字符串类型能存储的最大容量是512MB，其键也是512M

字典

不满足intset条件时

zset

zset中每个元素包含数据本身和一个对应的分数score

经典的例子：一个zset的key是“math”，代表数学课的成绩，然后可以网这个key里插入很多数据，每个数据包含一个姓名和一个成绩，那么姓名就是数据本身，成绩就是score

zset中数据不允许重复，分数可以重复

zset的底层实现：

  1. 数据少时，使用ziplist编码，条件为：

    • 有序集合保存的元素数量不超过128个
    • 有序集合保存的所有元素的成员长度都小于64字节

    ziplist编码的有序集合使用压缩列表作为底层实现。占用连续内存，每项元素都是（数据+score）的方式连续存储，按照score从小到大排序。ziplist为了节省内存，每个元素占用的空间可以不同，对于大的数据（long long），就多用一些字节来存储，而对于小的数据（short），就少用一些字节来存储。因此查找的时候需要按顺序遍历。ziplist省内存但是查找效率低。

  2. 数据多时，使用字典+跳表：其中dict保存了从成员到分值的映射关系，zsl 则按分值由小到大保存了所有的集合元素。这样，当按照成员来访问有序集合时可以直接从dict中取值，当按照分值的范围访问有序集合时可以直接从zsl中取值，采用了空间换时间的策略以提高访问效率。

    跳表式从有序链表中选取部分节点，组成一个新链表，并以此作为原始链表的一级索引，再从以及索引中选取部分节点，组成一个新的链表，作为二级索引。跳表在查找时优先从高层开始查找，若next节点大于目标值，或next执行null，则从当前节点下降一层继续向后查找。

redis使用跳表而不是红黑树的原因：

  • 按照区间查找数据这个操作，红黑树的效率没有跳表高。跳表可以在 O(logn)时间复杂度定位区间的起点，然后在原始链表中顺序向后查询就可以了。
  • 相比于红黑树，跳表还具有代码更容易实现、可读性好、不容易出错、更加灵活等优点。
  • 插入、删除时跳表只需要调整少数几个节点，红黑树需要颜色重涂和旋转，开销较大。

Redis与Memcached

两个都是非关系型内存键值数据库，主要有以下不同

  • 数据类型： Memcached 仅支持字符串和整数类型，而Redis支持五种不同类型的数据类型，更加灵活
  • 数据持久化：Redis 支持两种持久化策略：RDB快照和AOF日志，而memcached不支持持久化
  • 分布式：memcached不支持分布式，只能通过在客户端使用一致性哈希实现分布式存储，这种方式的存储和查询都需要现在客户端计算一次数据所在的节点
  • value值：Redis最大512M，Memcached最大只能1M
  • 内存管理机制：
    • 在Redis中并不是所有的数据都一直存储在内存中的，可以将一些很久没有使用的value交换到磁盘，而memcached的数据一直在内存中
    • Memcached将内存分割成特定长度的块来存储数据，以解决内存碎片问题。但这种方式会使内存的利用率不高

缓存

缓存分类

只读缓存：缓存只负责读，任何写操作都是直接操作在数据库上的，然后删除缓存的数据，然后下次在请求时，重新请求数据库 Redis

读写缓存：读和写都在缓存上，会在缓存上修改，需要考虑数据库宕机的情况，数据不一致

缓存写回数据库的时机有两种形式：一种是同步写回，另一种是异步写回

同步写回：缓存修改后，立刻修改数据库，这样可能降低响应速度，但这样可以保证安全性，需要事务来搞定。一起执行才行，不然就会产生数据不一致的情况

异步写回：缓存修改之后，找机会进行修改数据库。

缓存数据淘汰机制（策略）

可以设置内存最大使用量，当内存使用量超出时，会实施数据淘汰策略

lru：最近最少使用淘汰算法，也就是淘汰最长时间未被使用的

lfu：最不经常使用淘汰算法，也就是淘汰一定时期内访问次数最少的

Redis具有8种淘汰策略，其中有两种是redis4.0引入的：

Redis4.0 引入了volatitle-lfu 和allkeys-lfu淘汰策略，LFU策略通过统计访问频率，将访问频率最少的键值对淘汰

LRU：Redis会记录每个数据的最近一次访问的时间戳（LRU）。Redis执行写入操作时，若发现内存超出maxmemory，就会执行一次近似LRU淘汰算法。近似LRU会随机采样N个key，然后淘汰掉最旧的key，若淘汰后内存依然超出限制，则继续采样淘汰。可以通过maxmemory_samples配置项，设置近似LRU每次采样的数据个数，该配置项的默认值为5。

LFU（Least Frequently Used）是Redis4新增的淘汰策略，它根据key的最近访问频率进行淘汰。LFU在LRU的基础上，为每个数据增加了一个计数器，来统计这个数据的访问次数。当使用LFU策略淘汰数据时，首先会根据数据的访问次数进行筛选，把访问次数最低的数据淘汰出内存。如果两个数据的访问次数相同，LFU再比较这两个数据的访问时间，把访问时间更早的数据淘汰出内存。

如何处理淘汰的数据？

如果数据是干净的，则直接淘汰，如果数据是脏数据，则需要先写会数据库，在淘汰。不过Redis使用的是修改完之后，直接写入数据库，所以不会出现清理缓存的时候出现脏数据。

Redis的过期策略

Redis支持两种过期策略：

  • 惰性删除：客户端访问一个key的时候，Redis会先检查它的过期时间，如果发现过期会立刻删除这个key

  • 定期删除：Redis会将设置了过期时间的key放到一个独立的字典中，并对该字典进行每秒10的过期扫描

    过期扫描不会遍历该字典的所有key，使用简单地贪心策略：

    1. 从过期字典中随机选取20个key
    2. 删除这20个key中已过期的key
    3. 如果已过期的key比例超过25%，重复步骤1

缓存与数据库双写一致性问题

保证缓存与数据库双写一致，四种同步策略：

  1. 先更新缓存，再更新数据

  2. 先更新数据库，在更新缓存

  3. 先删除缓存，在更新数据库

  4. 先更新数据库，再删除缓存

更新缓存还是删除缓存？ 删除缓存更优

更新缓存：

  • 优点：每次数据变化都更新了缓存，所以查询时不容易出现没命中的情况
  • 缺点：更新缓存消耗比较大，如果数据经过复杂的计算再写入缓存，频繁的更新缓存，就会影响服务器的性能

删除缓存：

  • 优点：操作简单，无论更新操作是否复杂，都是将缓存中的数据删除
  • 缺点：删除缓存后，下一次查询会出现未命中的情况，这是需要重新读取一次数据库

先操作数据库还是缓存？

先更新数据库，再删除缓存，如果第二部出现失败的情况则可以采用重试机制解决问题

单线程环境下使用消息队列

将带修改的值存入消息队列，如果修改成功，则删除消息队列里的数据，如果失败，则从消息队列中取出，然后继续修改，就是重试

多线程环境下借助分布式锁redlock来解决问题

缓存穿透、缓存雪崩、缓存击穿

缓存雪崩

是指某一时刻缓存层无法继续提供服务，导致大量的请求直达数据库，引起数据库压力过大甚至宕机。可能是缓存中有大量的数据同时到期，或是redis节点发生故障。

解决方案：

  1. 设置过期时间时，附加一个随机数，这样就不会出现短时间内大量键过期
  2. 使用分布式数据库，分别请求不同而数据库，则会减轻某一个数据库的压力
  3. 设置热点数据永远不过期
  4. 启用降级，发生雪崩时，核心数据可以访问数据库，非核心数据直接返回预定的信息或空值或错误信息

缓存击穿

一份热点数据，他的访问量非常巨大，在其缓存失效的瞬间，大量请求直达数据库，导致数据库压力激增，无法处理请他请求

解决方案：

  1. 热点数据永不过期
  2. 加互斥锁：对数据的访问加互斥锁，当一个线程访问该数据时，其他线程只能等待，这个线程访问完后，缓存中的数据将会被重建，其他线程就可以在缓存中取值了

缓存穿透

客户端查询根本不存在的数据，使得请求直达数据库，导致其负载过大，甚至宕机。可能的原因：业务层误将缓存和数据库中的数据删除了，也有可能是有人恶意攻击，专门访问数据库中不存在的数据

解决方案：

  1. 缓存空对象：数据库未命中后，仍将空值存入缓存中，客户端再次访问时，缓存直接返回空值

  2. 布隆过滤器：将数据存入布隆过滤器，访问缓存之前过滤器拦截，若请求的数据不存在直接返回空值

  3. 前端过滤

    布隆过滤器原理：使用N个哈希函数，得出N个哈希值，并将对应的bit位置置为1。判断值是否存在，计算N个哈希值，对应的位是否存在0，存在0则表明该值不存在

    布隆过滤器的工作原理：

    1. 添加key时，每个哈希函数都利用这个key计算出一个哈希值，再根据哈希值计算一个位置，并将位数组中这个位置的值设置为1。
    2. 询问key时，每个哈希函数都利用这个key计算出一个哈希值，再根据哈希值计算一个位置。然后对比这些哈希函数在位数组中对应位置的数值：
      • 如果这几个位置中，有一个位置的值是0，就说明这个布隆过滤器中，不存在这个key。
      • 如果这几个位置中，所有位置的值都是1，就说明这个布隆过滤器中，极有可能存在这个key。之所以不是百分之百确定，是因为也可能是其他的key运算导致该位置为1。

缓存污染

含义：某些不被常访问的数据，白白浪费内存空间

解决方法：利用缓存淘汰策略，LFU策略，最不经常使用淘汰算法

LFU原理，使用一个字符串，前半段保存时间戳，后半段保存使用次数

Redis 集群方案

主从复制

主从复制模式，主库和从库的数据一致，采取的是读写分离的方式，进而降低Redis服务器压力

读操作：主和从都可以

写操作：先写主，然后主同步给从（防止数据不一致）

同步过程中，主服务器继续执行写操作，那么服务器则会漏掉这部分操作，所以主服务器在从服务器的同步过程中，会将新的写操作放到缓冲区中，从库同步完毕后，再执行缓冲区的文件，实现同步。

另外为了缓解主库同步压力，我们采取主-从-从的模式

哨兵集群

哨兵模型是主从复制模型的拓展，当主节点挂掉之后，从某个从节点顶替主节点。

通过哨兵节点来执行这个操作。用哨兵节点监控主节点，当发现主节点挂掉之后，选取某个从节点来替代主节点。

但是当哨兵节点挂掉后，整个监控系统也完蛋了，所以我们得搭建哨兵集群，当其中一个哨兵判断主节点挂掉后，询问其他节点该主节点是否挂掉，如果超过一定值的哨兵节点判断挂掉之后，则认定该节点已经挂掉。

Redis-Cluster 集群

redis 的哨兵模式基本已经可以实现高可用，读写分离，但是在这种模式下的每台redis服务器都存储相同的数据，很浪费内存，所以在redis3.0 上加入了cluster模式，实现redis的分布式存储，也就是说每台redis节点上存储着不同的内容。

持久化

Redis是内存型数据库，为了保证数据在断电后不会丢失，需要将内存中的数据持久化到硬盘上

三种持久化方式：RDB持久化、AOF持久化、RDB-AOF混合持久化

RDB持久化

RDB（Redis Database）是Redis默认的持久化方式，它是以快照的形式将进程中的数据持久化道硬盘上。RDB会创建一个压缩的二进制文件，文件以“.rdb”结尾，内部存储了数据库中键值对数据信息，RDB的触发方式有两种：

  • 手动触发：通过SAVE命令或是BGSAVE命令触发RDB持久化操作，创建“.rdb”文件
  • 自动触发：通过配置选项，让服务器在满足条件后自动执行BGSAVE命令。

其中，SAVE命令在执行期间，Redis服务器会被阻塞，直到“.rdb”文件创建完毕为止。而BGSAVE命令是异步版的SAVE命令，它会是Redis服务器创建一个子进程，创建”.rdb“文件。BGSAVE命令在创建子进程期间会存在短暂的阻塞，之后Redis服务器便可以继续处理客户端请求。总之BGSAVE命令是对SAVE命令的阻塞做出的优化，Redis内部所有设计RDB持久化的操作都是采用BGSAVE的方式。

思考一下，我们生成RDB的过程中，如果有数据发生了更改该怎么办呢？

fork 出的子进程用于生成RDB文件，然后在生成过程中，如果有写操作，则让主进程将这块数据复制一份，在复制份上进行写。

RDB持久化优缺点：

  • 优点：RDB生成压缩的二进制文件，体积小，使用改文件进行恢复数据时速度非常快
  • 缺点：BGSAVE命令每次执行都要fork操作创建一个子进程，属于重量级操作，不宜频繁操作，所以RDB持久化没法做到实时的持久化。

AOF持久化

AOF（append only file），解决了数据持久化的实时性，是目前Redis主流的持久化方式。AOF以独立日志的方式，记录每次的写入命令，重启时再重新执行AOF文件中的命令来恢复数据。

AOF的工作流程包括：命令写入（append）、文件同步（sync）、文件重写（rewrite）、重启加载（load）

AOF持久化的文件同步机制：

为了提高程序的写入性能，现代操作系统会把针对硬盘的多次写入优化为一次写入操作。

  1. 当程序调用write对文件写入时，系统不会把数据直接写入磁盘，而是将数据放到内存缓冲区里。
  2. 当达到特定的时间周期或缓冲期写满时，系统才会执行flush操作，将缓冲区的数据冲洗志磁盘

冲洗机制直接影响AOF持久化的安全性，Linux系统中glibc提供了fsync函数，可以将指定的文件强制从缓冲区刷到磁盘， redis向用户提供了appendfsync选项，来控制系统冲洗AOF的频率。

AOF持久化的优缺点：

优点：与RDB持久化可能丢失大量数据相比，AOF持久化安全性更高，通过everysec选项，用户可以将数据丢失的时间窗口限制在1秒内。

缺点：AOF文件存储的是协议文本，他的体积比二进制格式的“.rdb” 文件大很多。AOF需要执行AOF文件中的命令来恢复数据库，速度比RDB慢很多。AOF在进行重写时也需要创建子进程，在数据库体积较大时占用资源比较多，会导致服务器短暂的阻塞。

RDB-AOF混合持久化：

Redis 从4.0开始引入了RDB-AOF混合持久化模式，这种模式是基于AOF构建而来的，用户可通过配置文件中的“aof-use-rdb-preamble yes”来开启。

Redis服务器在执行AOF重写操作时，会按照如下原则处理数据：

  • 像执行BGSAVE命令一样，根据数据库当前的状态生成相应的RDB数据，并将其写入AOF文件
  • 对于重写之后执行的Redis命令，则以协议文本的方式追加到AOF文件的末尾，即RDB数据之后。

这样使用RDB-AOF混合持久化，同时获得RDB和AOF的优点，服务器既可以通过AOF文件包含的RDB数据来实现数据的快速恢复，又可以通过AOF将数据丢失的窗口限制在1秒内。

事务

Redis中事务不保证原子性，单条命令是原子执行的，但事务不保证原子性，且没有回滚。事务中任意命令执行失败，其余的命令仍会被执行。

Redis中的事务可以视为一个队列，我们可以通过MULTI开始一个事务，相当于声明了一个命令队列，接下来，我们向Redis中提交的每条命令，都会被排入这个命令队列，当输入EXEC命令时，将触发当前事务，以先进先出的方式从命令队列中取出这条命令并执行，所以Redis中的一个事务从开始到执行会经历 开始事务、命令入队和执行事务三个阶段。

事务的生命周期

开启事务（Redis 中使用 MULTI 来实现）

执行事务中的操作：增删改查等操作，Redis 会将这些操作保存在一个队列中，暂且先不执行

提交事务 EXEC 提交事务

Redis 的事务能保证哪些特性？

  1. 原子性
  • 命令入队时就报错，会放弃事务执行，保证原子性；命令入队时没报错，
  • 实际执行时报错，不保证原子性；
  • EXEC 命令执行时实例故障，如果开启了 AOF 日志，可以保证原子性。

  1. 一致性

    在事务开始之前和事务结束以后，数据库的完整性没有被破坏。这表示写入的资料必须完全符合所有的预设约束、触发器、级联回滚等

    Redis 事务满足一致性

  2. 隔离性

    如果开启了WATCH 机制则没有被破坏，因为其会监控键值对的改变。保证一致性

  3. 持久性

    不能保证持久性

分布式锁

什么时候需要？

在分布式环境下，当多个server并发修改同一个资源时，为了避免竞争就需要使用分布式锁。为什么不用Java自带的锁？Java中的锁是面向多线程设计的，他只局限与当前的JRE环境，而多个server实际上是多个进程，是不同的JRE环境，所以Java自带的锁机制在这个场景下是无效的。

Redis 如何实现分布式锁

采用Redis实现分布式锁，就是在Redis中存一份代表锁的数据，通常用字符串即可

  1. 加锁

    通过“set .. nx ..”命令来将加锁和过期时间编排到一起，保证了原子性，避免死锁

  2. 解锁

    解锁就是删除代表锁的数据

有个隐患，进程A在任务还没结束的时候，所已经到期释放了。等进程A结束，依然会释放锁，因为逻辑代码就是任务结束后释放锁。但是它的锁早就释放了，它此时释放的可能是其他进程的锁

解决方法：

  1. 加锁时就要给锁设置一个标志，进程要记住这个标志。当进程解锁的时候要先判断是否是自己持有的锁，是才能释放。可以给key赋一个随机值，来充当进程的标志
  2. 解锁时要先判断，在释放，这两步要保证原子性。这需要采用Lua脚本，将两个命令编排在一起

基于RedLock算法的分布式锁

上述分布式锁的方式是针对单个节点的情况，对于多个节点的情况，一般采用RedLock算法的实现方案。

基本逻辑：

  • 这些节点相互独立，不存在主从复制或集群协调机制
  • 加锁：以相同的key向n个实例加锁，只要超过半数成功了，则认定加锁成功
  • 解锁：向所有实例发送DEL命令，进行解锁

可以自己实现该算法，也可以直接使用Redission框架

其他面试题

多台Redis抗高并发访问该设么设计？（并发量高的时候，怎么设计Redis）

Redis cluster是Redis的分布式解决方案，有效地解决了Redis分布式方面的需求，当遇到单机内存、并发、流量瓶颈时，可一次啊用cluster架构方案达到负载均衡的目的。

Redis集群采用虚拟槽分区来实现数据的分片，他把所有的键根据哈希函数映射到0-16386整数槽内，每个节点负责维护一部分槽以及槽所映射的键值数据。

虚拟槽分区具有以下特点：

  1. 解耦数据与节点之间的关系，简化了节点扩容和收缩的难度
  2. 节点自身维护槽的映射关系，不需要客服端或代理服务维护槽分区数据
  3. 支持节点、槽、键之间的映射查询，用于数据路由、在线伸缩等场景

HyperLogLog

应用场景：统计一个网站的UV（独立的访客，每个用户每天只记录一次），怎么实现？

如果统计PV（浏览量，用户每点击一次或刷新一次就记录一次），给每个页面配置一个独立的redis计数器，把这个计数器的key后缀加上当天的日期，每次请求就执行一次increment操作，最终就可以统计出所有的PV了。

但是UV不同，每个用户一天之内多次访问请求只能计数一次。如果用HashMap或是Set这种结构，数据一旦达到百万或是千万量级会导致程序占用大量内存。存储量巨大，并且统计复杂。使用HyperLogLog

第一种B树

B树最大的优势是插入和查找的效率很高。如果用B树存储要统计的数据的话，可以快速判断新的数据是否存在，并快速将元素插入到B树中。要计算基础值，只需要计算B树的节点个数就可以啦。

不过将B树结构维护到内存中，能够解决统计和计算的问题，但是并没有节省内存。

第二种Bitmap

bitmap可以理解为通过一个bit数组来存储特定数据的一种数据结构，每一个bit位都能包含信息。bitmap还有一个优势就是可以轻松的合并多个统计结果，只需要对结果求异或就可以啦。1亿的数据量，用mapbit，大约需要的内存：100000000 / 8 / 1024 / 1024 = 12M，虽然bitmap节省空间，但仍然不够，统计一个对象的基数需要12M，如果统计一万个对象，就需要接近120G，对于大数据场景仍然不适用
